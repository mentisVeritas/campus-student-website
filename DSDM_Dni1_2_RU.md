# DSDM: выполнение заданий за День 1-2 (документная часть)

## 0) Рамки этапа
- Формат: только документная работа, без реализации кода.
- Период: День 1-2.
- Дедлайн проекта: 8 рабочих дней + 2 резервных дня.

## 1) День 1 — Контекст и рамки

### 1.1 Kickoff и договоренности по DSDM
- Работаем итерационно, релизный приоритет определяется через MoSCoW.
- Изменения требований допускаются в любой день, но фиксируются в журнале.
- Решения принимаются по правилу: сначала ценность для пользователя, потом сложность.
- Контрольные синки: короткие ревью каждые 2-3 дня.
- Резервные дни 9-10 используются только на качество (фиксы, стабилизация, регрессия).

### 1.2 Черновик бизнес-кейса (1 страница)
В университете критичная информация для студентов и абитуриентов фрагментирована: расписание, объявления, академические результаты и административные процессы находятся в разных каналах. Это снижает скорость коммуникации и увеличивает ручную нагрузку на сотрудников. Проект Campus Student Website должен стать единой цифровой точкой входа с понятным UX и приоритизацией функций по реальной ценности.

Деловая цель на первый релиз: к Дню открытых дверей показать стабильный прототип, который закрывает базовые академические и информационные сценарии (вход, профиль, расписание, оценки, новости, базовые запросы документов). Это повысит доверие к цифровой стратегии университета и сократит операционные издержки на рутинные запросы.

### 1.3 Критерии успеха этапа
- Согласованная таблица MoSCoW для всех 35 функций.
- Ясный план итераций на дни 3-8 с контрольными точками.
- Зафиксированные риски и правила изменения scope.

### 1.4 Грубая декомпозиция функций (черновая)
- `Must`: базовый доступ, академическое ядро, личный профиль.
- `Should`: функции, усиливающие ценность демо и админ-процессы.
- `Could`: полезные, но не критичные для дедлайна.
- `Won't`: тяжелые интеграции и высокорисковые соц/финтех сценарии.

### 2.2 План итераций на дни 3-8 и контрольные точки
- **Дни 3-5 (Must-итерация):** функции `1-6` из таблицы (ядро доступа и академический блок), базовый UX, сквозные переходы.
- **День 6 (Should-итерация):** функции `7-13` из таблицы + QA-волна 1.
- **День 7 (изменение требований):** обрабатываем внезапный запрос, перепланируем scope, фиксируем trade-offs.
- **День 8 (стабилизация):** freeze, багфиксы P1/P2, регрессия, RC.

Контрольные точки:
- Checkpoint A (конец Дня 2): согласован MoSCoW и риски.
- Checkpoint B (конец Дня 5): готов Must-инкремент.
- Checkpoint C (конец Дня 6): готовы Should-улучшения и bug-list.
- Checkpoint D (конец Дня 8): стабильный RC перед резервом.

### 2.3 Короткое ревью с преподавателем (фиксация)
**Комментарий преподавателя:**  
"Почему электронная зачетная книжка в Must? Даже базовая API-интеграция может занять несколько дней. Пересмотрите."

**Решение команды:**  
- Оставляем функцию в `Must` как ключевую ценность, но с ограничением объема.  
- Для MVP используем упрощенную реализацию (агрегированные учебные данные без сложной интеграции).  
- Риск интеграции переносим в post-MVP backlog с отдельной оценкой.
